01

Visão geral da arquitetura

O sistema é um pipeline de migração de código Informix para SQL Server (T-SQL) em aplicações C#/.NET legadas. Ele roda dentro do Claude Code, autenticado pela assinatura (sem consumo de créditos de API), e combina quatro peças que se complementam.

┌──────────────────────────────────────────────────────────────┐
│  CLAUDE CODE  (assinatura — sem billing)                     │
│                                                              │
│  CLAUDE.md ............... ORQUESTRADOR (identidade + fluxo)  │
│     └ seção 5.5: pipeline automático                         │
│                                                              │
│  Skill "regras-conversao-informix" .. FONTE ÚNICA de regras  │
│     (carrega sob demanda — progressive disclosure)           │
│                                                              │
│  Subagentes (.claude/agents/):                               │
│     ├ informix-analyzer   (Haiku)  .. triagem                │
│     ├ informix-converter  (Sonnet) .. conversão + RAG        │
│     └ informix-validator  (Sonnet) .. revisão independente   │
│                                                              │
│  RAG:  ChromaDB (Docker) ── MCP server ── ferramenta         │
│        "buscar_exemplos_migracao"                            │
└──────────────────────────────────────────────────────────────┘

FLUXO:  "migre X"  →  analyzer  →  converter  →  validator  →  consolida

O papel de cada peça

PeçaResponsabilidadeOnde vive
CLAUDE.mdIdentidade do agente + orquestração do pipeline. Não contém regras técnicas nem consulta RAG.Raiz do projeto
SkillTodas as regras técnicas de conversão (funções, tipos, schemas, DataReader). Fonte única, carregada só quando necessária..claude/skills/
analyzerTriagem barata: o arquivo tem Informix? o quê? Modelo leve (Haiku)..claude/agents/
converterAplica as conversões. Usa o Skill + consulta o RAG. Modelo forte (Sonnet)..claude/agents/
validatorRevisão independente da conversão, contra o Skill. Modelo forte (Sonnet)..claude/agents/
RAGMemória de exemplos de conversão já validados. Busca por similaridade.ChromaDB (Docker) + MCP
Princípio de design

CLAUDE.md = quem o agente é e como responde (identidade). Skill = como converter cada construção (conhecimento técnico). Subagentes = execução isolada por especialidade. RAG = experiência acumulada. Separar essas camadas mantém cada parte enxuta e substituível.

06

O CLAUDE.md (orquestrador)

O CLAUDE.md fica na raiz do projeto e é lido automaticamente pelo Claude Code em toda sessão. Ele define a identidade do agente e a orquestração — não contém as regras técnicas (essas vivem no Skill) nem consulta o RAG (isso é do converter).

O que o CLAUDE.md contém

  • Papel e contexto: especialista em migração Informix → SQL Server em C#/.NET legado.
  • Escopo estrito: só migração, nunca manutenção/refatoração/clean code.
  • Anti-alucinação: nunca inventar tabelas, colunas, schemas; quando faltar dado, declarar pendência.
  • Formato de resposta padronizado (Resumo → Validados → Inconsistências → Pendências → Conclusão).
  • Seção 5.5 — Orquestração: o fluxo do pipeline (ver seção 10 deste guia).
  • Referência ao Skill: aponta para regras-conversao-informix como fonte das regras técnicas.
Divisão de responsabilidades

Regra de ouro: se o conteúdo descreve quem o agente é / como ele responde → CLAUDE.md. Se descreve como converter uma construção específica → Skill. Se é um exemplo concreto já resolvido → RAG.

Como recriar o conteúdo

O CLAUDE.md completo (≈310 linhas) acompanha este guia como arquivo separado. Copie-o para a raiz do projeto. Alternativamente, você pode pedir ao próprio Claude Code para gerá-lo a partir da estrutura descrita aqui e depois revisar.

Após colocar o arquivo, reinicie o Claude Code e teste que ele carregou:

CLAUDE CODE
Segundo as instruções deste projeto, qual é o seu escopo
e o que você NÃO deve fazer?

Ele deve responder mencionando "migração Informix → SQL Server", "não misturar com manutenção" e "não alterar regra de negócio".

O Skill (regras técnicas)

O Skill é onde vivem todas as regras técnicas de conversão. A vantagem: ele carrega sob demanda (progressive disclosure) — só entra no contexto quando uma conversão realmente acontece, mantendo o CLAUDE.md enxuto.

Localização e estrutura

POWERSHELL
New-Item -Path .claude\skills\regras-conversao-informix `
  -ItemType Directory -Force
# o arquivo vai em:
# .claude\skills\regras-conversao-informix\SKILL.md

Anatomia do SKILL.md

MARKDOWNSKILL.md (frontmatter)
---
name: regras-conversao-informix
description: Regras técnicas de conversão de Informix
  para SQL Server (T-SQL) em C#/.NET. Use ao analisar,
  converter ou revisar SQL, DAO, ADO.NET, iBATIS ou EF...
---

# corpo: as regras de transformação

O corpo do Skill cobre, em seções: funções SQL (NVL→ISNULL, TODAY, CURRENT, MATCHES, SKIP/FIRST, ||), objetos ADO.NET, tipos de dados e a armadilha BIT vs CHAR, schemas, recuperação de identidade (SCOPE_IDENTITY e o cuidado com numeric(38,0)), datas, iBATIS/MyBatis, Entity Framework, e o padrão crítico de leitura do DataReader.

A description é crucial

O campo description é o que o Claude usa para decidir quando carregar o Skill. Escreva-a listando os gatilhos ("use ao analisar/converter SQL Informix, DAO, ADO.NET..."). Uma boa description garante que o Skill seja acionado na hora certa.

Reinício necessário

Ao criar o diretório .claude/skills/ pela primeira vez, o Claude Code precisa reiniciar para detectá-lo (uma sessão em execução não vê diretórios criados depois do início).

Teste: peça uma análise que exija regras. Na resposta deve aparecer Skill(regras-conversao-informix) - Successfully loaded skill.

08

O RAG: indexação e MCP server

O RAG (Retrieval-Augmented Generation) é a memória de exemplos do projeto. Cada conversão validada vira um exemplo vetorizado no ChromaDB; na hora de converter, o agente busca os mais parecidos. São três componentes: os exemplos (JSON), o indexador e o MCP server.

8.1 — Formato dos exemplos

Cada exemplo é um JSON em rag\knowledge_base\examples\:

JSONexemplo-semente
{
  "id": "seed_001_nvl_schema",
  "categoria": "funcao_sql",
  "tags": ["NVL", "ISNULL", "schema"],
  "source": "SELECT NVL(nome, 'SEM NOME') FROM informix.cliente",
  "target": "SELECT ISNULL(nome, 'SEM NOME') FROM dbo.cliente",
  "notes": "NVL é do Informix; usar ISNULL. Schema validar."
}

8.2 — O indexador

Crie rag\indexer.py. Ele lê os JSONs, gera embeddings localmente (sem billing) e grava no ChromaDB:

PYTHONrag/indexer.py (essência)
import json
from pathlib import Path
import chromadb
from sentence_transformers import SentenceTransformer
from config import settings

def main():
    model = SentenceTransformer(settings.EMBEDDING_MODEL)
    client = chromadb.HttpClient(host=settings.CHROMA_HOST,
                                 port=settings.CHROMA_PORT)
    col = client.get_or_create_collection("migration_knowledge")

    files = list(Path("rag/knowledge_base/examples").glob("*.json"))
    ids, docs, metas = [], [], []
    for f in files:
        ex = json.load(open(f, encoding="utf-8"))
        texto = f"ORIGEM:\n{ex['source']}\n\nDESTINO:\n{ex['target']}\n\nNOTAS:\n{ex.get('notes','')}"
        ids.append(ex["id"]); docs.append(texto)
        metas.append({"categoria": ex.get("categoria","")})

    emb = model.encode(docs).tolist()
    col.upsert(ids=ids, documents=docs, embeddings=emb, metadatas=metas)
    print(f"OK: {len(ids)} exemplos. Total: {col.count()}")

if __name__ == "__main__":
    main()

Rode como módulo (para os imports funcionarem):

POWERSHELL
python -m rag.indexer   # baixa o modelo (1ª vez) e indexa

8.3 — O MCP server

O MCP server expõe a busca no ChromaDB como uma ferramenta que o Claude Code pode chamar. Crie mcp_servers\rag_server.py.

Regra crítica do MCP: stdout é sagrado

O protocolo MCP usa o stdout para trafegar JSON. Nenhum print() e nenhum log de biblioteca pode ir para o stdout, ou o handshake quebra ("Invalid JSON"). Por isso o server silencia os logs e usa lazy loading (carrega o modelo só na primeira busca), o que também evita timeout na subida.

PYTHONmcp_servers/rag_server.py (essência)
import os, logging
os.environ["TRANSFORMERS_VERBOSITY"] = "error"
os.environ["HF_HUB_DISABLE_PROGRESS_BARS"] = "1"
logging.disable(logging.WARNING)   # silencia libs ANTES de importar

import chromadb
from sentence_transformers import SentenceTransformer
from mcp.server.fastmcp import FastMCP
from config import settings

mcp = FastMCP("rag-migracao-informix")
_model = _col = None            # lazy loading

def _ready():
    global _model, _col
    if _model is None:
        _model = SentenceTransformer(settings.EMBEDDING_MODEL)
        c = chromadb.HttpClient(host=settings.CHROMA_HOST, port=settings.CHROMA_PORT)
        _col = c.get_or_create_collection("migration_knowledge")

@mcp.tool()
def buscar_exemplos_migracao(trecho_codigo: str, n_resultados: int = 3) -> str:
    """Busca exemplos de conversão Informix→SQL Server similares ao trecho."""
    _ready()
    if _col.count() == 0:
        return "Base RAG vazia."
    n = min(n_resultados, _col.count())
    emb = _model.encode([trecho_codigo]).tolist()
    r = _col.query(query_embeddings=emb, n_results=n,
                   include=["documents","metadatas","distances"])

Os três subagentes

Subagentes são arquivos .md em .claude\agents\. Cada um roda em contexto isolado, com seu próprio modelo e ferramentas. Isso mantém o trabalho pesado (leitura de arquivos, RAG) fora do contexto do principal.

9.1 — analyzer (triagem)

Classifica em PRECISA_MIGRACAO / JA_SQLSERVER / SEM_BANCO e cataloga o que muda, em alto nível. Não converte. Modelo leve (Haiku) porque é tarefa de alto volume. Saída curta e estruturada, que alimenta a decisão de rota do principal.

9.2 — converter (conversão)

Aplica as conversões. No fluxo dele: (1) consulta o RAG via buscar_exemplos_migracao; (2) usa o Skill regras-conversao-informix como fonte das regras; (3) produz a conversão com mínima intervenção. A ferramenta MCP é declarada assim no frontmatter:

YAML
tools: Read, mcp__rag-migracao__buscar_exemplos_migracao

9.3 — validator (revisão independente)

Recebe uma conversão pronta e a revisa contra o Skill, procurando: transformações omitidas, conversões incorretas, regra de negócio alterada, schema assumido sem ressalva, pendências não registradas. Emite veredito APROVADA / COM RESSALVAS / REPROVADA. Não usa RAG — revisa contra as regras canônicas, garantindo independência do converter.

Por que o validator não usa RAG

O papel dele é checar contra as regras (fonte canônica), não contra exemplos. Se usasse o mesmo RAG do converter, poderia herdar o mesmo viés e concordar com o mesmo erro. Revisar só contra o Skill torna a revisão genuinamente independente.

Reinício + verificação

Após criar os arquivos, reinicie o Claude Code. O comando /agents (wizard) foi removido em versões recentes — teste invocando diretamente: "Use o subagente informix-analyzer para triar este trecho: ...". Para ver as chamadas internas de um subagente em background, use Ctrl+O para expandir.

Limitação de MCP em subagentes

Versões antigas do Claude Code não davam acesso a MCP dentro de subagentes. Nas recentes (testado em v2.1.206) funciona — o converter consulta o RAG sozinho. Se num ambiente o converter não conseguir usar o RAG, o plano B é deixar o RAG no principal, que consulta e repassa os exemplos ao delegar.

10

Orquestração e teste do pipeline

A seção 5.5 do CLAUDE.md ensina o principal a encadear os três subagentes automaticamente quando você pede uma migração. O fluxo:

Pedido: "migre este arquivo/trecho"
   │
   ├─1─▶ informix-analyzer  (triagem)
   │       └─ PRECISA_MIGRACAO | JA_SQLSERVER | SEM_BANCO
   │
   ├─2─▶ decisão de rota:
   │       JA_SQLSERVER / SEM_BANCO  →  PARA. Reporta "nada a migrar".
   │       PRECISA_MIGRACAO          →  prossegue.
   │
   ├─3─▶ informix-converter  (converte: RAG + Skill)
   │
   ├─4─▶ informix-validator  (revisa a conversão)
   │       └─ APROVADA | COM RESSALVAS | REPROVADA
   │
   └─5─▶ principal CONSOLIDA:
           APROVADA          →  apresenta conversão + ressalvas
           REPROVADA         →  reporta problema e correção, e PARA
                                (você decide — sem re-conversão automática)

Regras de orquestração (resumo)

  • Parada antecipada: se não há nada a migrar, não chama converter nem validator.
  • Reprovação transparente: se o validator reprova, reporta e para — você decide o próximo passo (sem loop automático nesta versão).
  • Invocação individual continua permitida: "só faça a triagem", "use o validator para revisar isto".

Teste do pipeline completo

Confirme o MCP (/mcp → connected) e peça uma migração sem invocar subagentes manualmente:

CLAUDE CODE
Migre este trecho:

SELECT FIRST 20 NVL(cliente, 'N/A'), saldo || ' reais'
FROM informix.conta WHERE atualizado = TODAY

Você deve ver a sequência automática: ● informix-analyzer → ● informix-converter → ● informix-validator → consolidação.

Expectativa de tempo

O pipeline completo (Haiku + 2× Sonnet em sequência) leva alguns minutos por trecho. Para um trecho isolado parece caro; o valor aparece na escala — o analyzer barato filtra muitos arquivos e só os que precisam pagam o custo do pipeline. É o padrão que habilita processar pastas inteiras.

2

Transforme cada conversão validada num novo JSON em rag\knowledge_base\examples\ (source, target, notes).

3

Rode python -m rag.indexer para re-indexar. Os novos exemplos passam a ser consultados.

4

Com o tempo, a similaridade melhora e as conversões ficam mais consistentes com o padrão do projeto.

Versionar tudo com Git

O projeto inteiro deve estar sob controle de versão — é o que permite replicar e compartilhar. Crie um .gitignore excluindo venv/, data/chroma/ e __pycache__/. Versione: CLAUDE.md, .claude/ (skills e agents), mcp_servers/, rag/ (indexer + exemplos), config/, docker-compose.yml e este guia.

Para replicar em outra máquina

Clone o repositório e siga as seções 02 a 09 deste guia. Os arquivos de configuração (CLAUDE.md, SKILL.md, subagentes, rag_server.py, exemplos) já vêm no repositório — resta instalar os pré-requisitos, subir o Docker, indexar o RAG (python -m rag.indexer), registrar o MCP e logar no Claude Code.
